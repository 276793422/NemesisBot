# Peer Chat 超时配置优化

**修改日期**: 2026-03-05
**修改原因**: 支持长时间运行的 Agent/LLM 任务

## 🎯 修改内容

### 修改前（原配置）

```
┌─────────────────────────────────────────────────┐
│ RPC Client (本端)          │ 30 秒          │
├─────────────────────────────────────────────────┤
│ peer_chat handler (对端)  │ 60 秒          │
├─────────────────────────────────────────────────┤
│ RPCChannel RequestTimeout │ 60 秒          │
└─────────────────────────────────────────────────┘

问题：
- 30 秒对于复杂的 Agent 任务太短
- 容易超时失败
- 无法支持长时间 LLM 处理
```

### 修改后（新配置）

```
┌─────────────────────────────────────────────────┐
│ RPC Client (本端)          │ 30 分钟        │
│                          │ 1800 秒         │
├─────────────────────────────────────────────────┤
│ peer_chat handler (对端)  │ 29 分钟        │
│                          │ 1740 秒         │
├─────────────────────────────────────────────────┤
│ RPCChannel RequestTimeout │ 28 分钟        │
│                          │ 1680 秒         │
└─────────────────────────────────────────────────┘

特点：
- 分层超时保护机制
- 每层都比下一层略短
- 主动检测和报告超时
- 支持长时间 Agent 任务
```

## 📝 详细修改列表

### 修改 1: RPC Client 超时（本端）

**文件**: `module/cluster/rpc/client.go`
**行数**: 194

**修改前**:
```go
timeout: 30 * time.Second,
```

**修改后**:
```go
timeout: 30 * time.Minute, // 30 minutes for long-running LLM tasks
```

**说明**:
- 本端等待对端响应的总时间
- 从 30 秒增加到 30 分钟（60倍）
- 覆盖大部分 Agent 任务的处理时间

---

### 修改 2: peer_chat handler 超时（对端）

**文件**: `module/cluster/rpc/peer_chat_handler.go`
**行数**: 117

**修改前**:
```go
ctx, cancel := context.WithTimeout(context.Background(), 60*time.Second)
```

**修改后**:
```go
ctx, cancel := context.WithTimeout(context.Background(), 29*time.Minute)
```

**说明**:
- 对端 peer_chat handler 等待 LLM 响应的时间
- 从 60 秒增加到 29 分钟（29倍）
- 比本端超时略短（29 < 30），让对端主动返回错误

---

### 修改 3: RPCChannel RequestTimeout

**文件**: `module/agent/loop.go`
**行数**: 1593

**修改前**:
```go
RequestTimeout: 60 * time.Second,
```

**修改后**:
```go
RequestTimeout: 28 * time.Minute,
```

**说明**:
- RPCChannel 等待 MessageBus 响应的时间
- 从 60 秒增加到 28 分钟（28倍）
- 只针对 RPC 通道有效（其他通道不受影响）
- 比对端 handler 略短（28 < 29），作为最内层保护

## 🎯 分层超时机制

### 超时检测顺序（从快到慢）

```
1. RPCChannel (28 分钟)
   ├─ 超时 → 清理 pending request
   └─ 返回超时

2. peer_chat handler (29 分钟)
   ├─ 超时 → 返回错误响应
   └─ 通知 RPCChannel

3. RPC Client (30 分钟)
   ├─ 超时 → 本端返回超时错误
   └─ 用户看到 "timeout waiting for response"
```

### 好处

1. **主动错误报告**: 每层都能主动检测超时
2. **清晰错误来源**: 知道是哪一层超时
3. **资源保护**: 及时清理，避免资源泄漏
4. **用户友好**: 明确的错误信息
5. **长任务支持**: 支持 Agent 执行长时间任务

## 💡 设计亮点

### 1. 分层超时保护

每一层都有独立的超时检测，形成保护链：
```
内层 → 外层
RPCChannel → peer_chat handler → RPC Client
  28分钟      →    29分钟      →   30分钟
```

### 2. 合理的超时梯度

```
28分钟 < 29分钟 < 30分钟
  ↓          ↓          ↓
 内层      中间      外层
```

### 3. 责任明确

- **RPCChannel**: 清理自己的 pending request
- **peer_chat handler**: 返回明确的错误信息
- **RPC Client**: 最终的超时保护

### 4. 用户友好

- 不需要猜测哪里出问题
- 每层都有明确的日志
- 错误信息清晰
